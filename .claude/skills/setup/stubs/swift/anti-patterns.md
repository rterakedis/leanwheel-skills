# Anti-Patterns & AI Guardrails

> Updated: 2026-07-19 — iOS 18+
> Patterns frequently generated by AI from legacy training data. Each is a **hard rejection** in code review — do not approve PRs containing these.

---

## 1. ViewModel Classes for Single Views

**Symptom:** A file named `*ViewModel.swift` that exists solely to serve one `*View.swift`.

**Why it's wrong:** SwiftUI views are already the ViewModel. `@State`, computed properties, and `@FetchRequest` in the View struct handle everything these classes do — without the ceremony, the extra file, or the `@Published` overhead.

```swift
// ❌ Banned
@Observable final class ProfileViewModel {
    var name: String = ""
    var email: String = ""
    var isLoading: Bool = false
    func load() async { ... }
}

// ✅ Replace with
struct ProfileView: View {
    @State private var name = ""
    @State private var email = ""
    @State private var isLoading = false

    var body: some View { ... }
}
```

## 2. Legacy Observation (`ObservableObject` / `@Published`)

**Symptom:** Any of: `ObservableObject`, `@Published`, `@StateObject`, `@ObservedObject`, `@EnvironmentObject`.

**Why it's wrong:** These are iOS 14–16 patterns. The `@Observable` macro (iOS 17+) replaces all of them with less code, finer-grained re-renders, and no boilerplate. Legacy observation triggers full view re-renders on any `@Published` change; `@Observable` only re-renders views that actually read the changed property.

```swift
// ❌ Banned
class OrderStore: ObservableObject {
    @Published var orders: [Order] = []
}

// ✅ Replace with
@Observable final class OrderStore {
    var orders: [Order] = []
}
```

## 3. `.onAppear { Task { } }` for Async Work

**Symptom:** `.onAppear { Task { await something() } }`

**Why it's wrong:** The `Task` is not cancelled when the view disappears. If the user navigates away before the task completes, it continues running — leaking work and potentially mutating state for a view that no longer exists.

```swift
// ❌ Banned — leaks on dismiss
.onAppear {
    Task { await orderService.refresh() }
}

// ✅ Replace with
.task { await orderService.refresh() }
```

## 4. `DispatchQueue` in New Service Code

**Symptom:** `DispatchQueue(label:)`, `DispatchQueue.main.async`, `DispatchQueue.global()` in any file created after Swift Concurrency adoption.

**Why it's wrong:** `DispatchQueue` has no compiler-enforced isolation. Swift `actor` and `@MainActor` give the same guarantees with compile-time checking and full integration with `async/await`.

```swift
// ❌ Banned
private let queue = DispatchQueue(label: "sync", qos: .utility)
queue.async { self.sync() }
DispatchQueue.main.async { self.isLoading = false }

// ✅ Replace with
actor SyncEngine { func sync() async { ... } }
@MainActor func finishLoading() { isLoading = false }
```

---

## 5. `import Combine` in New Files

**Symptom:** `import Combine` in any file that isn't wrapping an existing Combine-based API for backwards compatibility.

**Why it's wrong:** Every Combine pattern used in SwiftUI has a direct `async/await` equivalent that is simpler, more readable, and better integrated with Swift Concurrency.

| Combine pattern | Replacement |
|---|---|
| `publisher.sink { }` with `@Published` | `@Observable` + `.task` |
| `PassthroughSubject` for events | `AsyncStream` or `async func` |
| `Debounce` on search | `.task(id: searchText)` with `Task.sleep` |
| `combineLatest` | `async let` or `TaskGroup` |

## 6. `Array(fetchResults)` Passed from Parent to Child

**Symptom:** Converting `@FetchRequest` results to an `Array` and passing it to a child view as a parameter.

**Why it's wrong:** `@FetchRequest` in a parent view drives re-renders when Core Data changes. Converting to `Array` and passing it down breaks this — the child view receives a snapshot, not a live binding, and won't update correctly. Each view that needs fetched data should own its `@FetchRequest`.

```swift
// ❌ Banned — child receives a dead snapshot
struct ParentView: View {
    @FetchRequest(sortDescriptors: []) var customers: FetchedResults<Customer>
    var body: some View {
        ChildView(customers: Array(customers))   // snapshot, not reactive
    }
}

// ✅ Child owns its own @FetchRequest
struct CustomerListView: View {
    @FetchRequest(sortDescriptors: [SortDescriptor(\.name)])
    private var customers: FetchedResults<Customer>
}
```

## 7. UIKit Lifecycle Mixing

**Symptom:** `UIViewControllerRepresentable` wrapping a screen that could be a SwiftUI view; `UIViewController` subclasses for anything that isn't a system integration point (camera, document picker, etc.).

**Why it's wrong:** Mixing UIKit lifecycles into SwiftUI navigation introduces state management split-brain. SwiftUI cannot observe `UIViewController` state, and UIKit cannot observe `@Observable` stores without fragile bridging code.

Use `UIViewRepresentable` / `UIViewControllerRepresentable` only for wrapping system-provided UIKit components with no SwiftUI equivalent.

## 8. Old `TabView` Content Embedding (iOS 18+)

**Symptom:** Placing view content directly in `TabView` with `.tabItem { }` instead of using the `Tab` wrapper.

**Why it's wrong (iOS 18+):** The old pattern doesn't support `.sidebarAdaptable` style, `.role(.search)` for search tabs, or `tabBarMinimizeBehavior`. It also doesn't adapt to sidebar layout on iPad automatically.

```swift
// ❌ Old pattern — misses iPad sidebar adaptation
TabView {
    ScheduleView()
        .tabItem { Label("Schedule", systemImage: "calendar") }
    CustomersView()
        .tabItem { Label("Customers", systemImage: "person.2") }
}

// ✅ iOS 18+ — Tab wrapper with sidebar adaptation
TabView(selection: $selectedTab) {
    Tab("Schedule", systemImage: "calendar", value: "schedule") {
        ScheduleView()
    }
    Tab("Customers", systemImage: "person.2", value: "customers") {
        CustomersView()
    }
}
.tabViewStyle(.sidebarAdaptable)
```

## 9. Manual Scroll Offset Detection (iOS 18+)

**Symptom:** Invisible sentinel views at the bottom of a list, `PreferenceKey` offset tracking, or `GeometryReader` used to detect scroll position.

**Why it's wrong (iOS 18+):** iOS 18 provides `onScrollPhaseChange`, `onScrollGeometryChange`, and `onScrollTargetVisibilityChange` for all of these use cases. The sentinel-view pattern is fragile, the `PreferenceKey` approach is verbose, and `GeometryReader` has layout side effects.

```swift
// ❌ Banned — invisible sentinel view for pagination
struct PaginationTrigger: View {
    var onAppear: () -> Void
    var body: some View { Color.clear.frame(height: 1).onAppear { onAppear() } }
}

// ✅ iOS 18+ — declarative visibility tracking
.onScrollTargetVisibilityChange(idType: Item.ID.self) { visibleIDs in
    if visibleIDs.last == items.last?.id { loadNextPage() }
}
```

## 10. `ForEach` with Index-Based Range on Mutable Data

**Symptom:** `ForEach(0..<items.count)` or `ForEach(items.indices)` for a mutable array.

**Why it's wrong:** When items are added or removed, index-based `ForEach` breaks SwiftUI's identity tracking — animations are wrong, state is misassigned to the wrong row, and the app may crash with index-out-of-bounds during list updates.

```swift
// ❌ Banned — breaks identity on any mutation
ForEach(0..<items.count, id: \.self) { index in
    ItemRow(item: items[index])
}

// ✅ Identity-based — stable across insertions and deletions
ForEach(items) { item in    // items: [Item] where Item: Identifiable
    ItemRow(item: item)
}
```

## 11. Stateful Persistence Logic Inside View-Lifecycle Modifiers

**Symptom:** Multi-step Core Data / persistence mutations (delete + cascade + regenerate, status transitions, chain repair, "fix-up the related rows") written inline inside `.onDisappear`, `.onChange`, `.task`, or a button's closure.

**Why it's wrong:** Logic buried in a SwiftUI view is **unreachable by unit tests**, so every regression ships silently. View-lifecycle timing (when exactly `onDisappear` fires, what objects are still valid) becomes load-bearing for correctness, and the same mutation gets copy-pasted across call sites and drifts. This is the single most common source of "we keep re-breaking this" persistence bugs.

The view's job is to capture **intent** and dispatch. The mutation belongs in a service that can be tested in isolation against an in-memory store.

```swift
// ❌ Banned — fetch/mutate/save buried in the view; untestable, regresses silently
.onDisappear {
    if let groupID = groupToCancel {
        let req = Job.fetchRequest()
        req.predicate = NSPredicate(format: "groupID == %@ AND status == %@", ...)
        let jobs = (try? context.fetch(req)) ?? []
        jobs.forEach { context.delete($0) }
        // ...20 more lines of pointer fix-up...
        try? context.save()
    }
}

// ✅ View captures intent and dispatches; service owns fetch → mutate → save
.onDisappear {
    if let groupID = groupToCancel {
        RecurrenceService.endService(groupID: groupID, context: context)
    }
    groupToCancel = nil
}

// ...and the logic is now unit-testable:
@Test func endService_deletesAllScheduledIncludingPastDated() throws { ... }
```

**Code-review rule:** any `context.delete` / `context.save` / multi-row fetch-and-mutate inside a view modifier is a rejection — extract to a `static`/`actor` service method and add a test. (See *Asserting deletion* in `testing.md` for how to test the extracted method.)

## 12. God Views & Data-Owning Sub-Views

**Symptom:** A single view/sheet file that has grown past ~300–400 lines because it owns status logic, permission gates, multiple sheet states, *and* all sub-view layout. Or — the failure mode while splitting it up — a sub-view struct that takes a service, `context`, or `@FetchRequest` so it can fetch/compute its own data.

**Why it's wrong:** An oversized view is hard for both humans and AI agents to edit safely — an agent changing one button must load and reason over everything else in the file, so every edit risks collateral damage. The *wrong* fix is to give an extracted sub-view its own data access; that quietly recreates a ViewModel (banned, see #1) and re-buries logic where tests can't reach it (see #11).

The correct decomposition uses two no-new-layer techniques: split type **members** into `extension TypeName {}` files grouped by responsibility (`+StatusActions`, `+MileageGate`), and split cohesive **layout** sections into named `private struct` sub-views. Sub-views receive only display data via `let`/`@Binding` and report actions up via closures — never a service or context down.

```swift
// ❌ Banned — sub-view fetches its own data; this is a ViewModel in disguise
private struct JobStatusButtons: View {
    @FetchRequest var history: FetchedResults<StatusChange>   // wrong — data access in a sub-view
    let context: NSManagedObjectContext                        // wrong — context passed down
    var body: some View { ... }
}

// ✅ Parent owns data; sub-view gets values down, sends intent up
private struct JobStatusButtons: View {
    let status: JobStatus
    let onAdvance: (JobStatus) -> Void
    var body: some View { ... }
}
```

**Code-review rule:** a view/sheet file over ~400 lines is a finding — decompose by responsibility (`extension` files + named sub-views), not by mechanical line-cutting. A sub-view struct that holds a service, `NSManagedObjectContext`, or `@FetchRequest` is a rejection. (See *File-Level Decomposition* in `ui-composition.md`.)

## 13. Deprecated & Superseded API Calls

**Symptom:** Any API in the left column — all still compile, all are legacy training-data artifacts.

| ❌ Legacy | ✅ Replacement |
|---|---|
| `foregroundColor(_:)` | `foregroundStyle(_:)` |
| `cornerRadius(_:)` | `clipShape(.rect(cornerRadius:))` |
| `.onChange(of:) { newValue in }` (1-param) | 2-param `{ old, new in }` or 0-param variant |
| `.navigationBarLeading` / `.navigationBarTrailing` | `.topBarLeading` / `.topBarTrailing` |
| `ScrollView(showsIndicators: false)` | `.scrollIndicators(.hidden)` |
| `overlay(SomeView(), alignment: .top)` | `overlay(alignment: .top) { SomeView() }` |
| `PreviewProvider` | `#Preview` macro |
| `UIGraphicsImageRenderer` for SwiftUI views | `ImageRenderer` |
| `UIImpactFeedbackGenerator` | `.sensoryFeedback(_:trigger:)` |
| `Text("a") + Text("b")` concatenation | Interpolation: `Text("a \(styledValue)")` |
| `WKWebView` in a representable (iOS 26+) | native SwiftUI `WebView` (`import WebKit`) |
| `Circle()` / `BorderedProminentButtonStyle()` as arguments | static member lookup: `.circle`, `.borderedProminent` |
| `TabView(selection:)` bound to `Int`/`String` | bind to a dedicated enum |
| `onTapGesture` on tappable content | `Button` (a11y + highlight; `onTapGesture` only when you need location/count) |

## 14. Legacy Foundation & C-Style Idioms

**Symptom:** Foundation patterns from the Objective-C era in new Swift code.

```swift
// ❌ Banned                                    // ✅ Replacement
DateFormatter() / NumberFormatter()             myDate.formatted(date: .abbreviated, time: .shortened)
Text(String(format: "%.2f", value))             Text(value, format: .number.precision(.fractionLength(2)))
str.replacingOccurrences(of: "a", with: "b")    str.replacing("a", with: "b")
name.contains(searchText)  // user filtering    name.localizedStandardContains(searchText)
items.filter { $0.isDone }.count                items.count(where: \.isDone)
Task.sleep(nanoseconds: 400_000_000)            Task.sleep(for: .milliseconds(400))
FileManager…urls(for: .documentDirectory…)      URL.documentsDirectory / url.appending(path:)
Date()  // "now" semantics                      Date.now
```

Also rejections: force-unwrap / `try!` outside genuinely unrecoverable paths, and **swallowed errors** — `catch { print(error) }` or `try?` that discards a failure the user should see. Errors surface to `@State` and the UI, or propagate.

## 15. Performance Smells — Observation Fan-Out & Main-Thread Image Decode

**Symptom (fan-out):** Many sibling views all reading one broad `@Observable` store (or one large array property on it), so any mutation invalidates everything.

**Why it's wrong:** `@Observable` is property-grained, but if every view reads the same array property, every change re-renders all of them. Narrow the observable surface: split stores per feature, or pass each row only its own item as a plain value.

**Symptom (images):** `UIImage(data:)` / decode / downsample inline in `body` or a row view.

**Why it's wrong:** Decoding happens on the main actor mid-scroll and drops frames. Decode and downsample off the main actor (`@concurrent` helper — see concurrency.md), cache the result, and hand views a ready-to-render image.
